# Changelog ‚Äî Project Caravela
# REQ-037.2: Record all ad hoc changes and deviations during implementation.
# Format: Date | Component | Change | Reason

| 2026-03-11 | `date_key` type | Confirmed `DATE` type for `dim_date.date_key` and all FK references in `fct_sales`, `fct_payments`, `fct_reviews` | `dbt_utils.date_spine` produces DATE natively ‚Äî no extra casting in `dim_date`. Staging models cast source timestamps via `DATE(CAST(... AS TIMESTAMP))`. Same-type FK joins across all fact tables. INTEGER (YYYYMMDD) would require additional casting in both `dim_date` and staging with no BigQuery benefit. REQ-008.1 Open Item (3) and REQ-017.1 blocking flag closed. |
| 2026-03-11 | BigQuery datasets | Renamed `raw` ‚Üí `olist_raw`; `analytics` ‚Üí `olist_analytics` | `raw` is a reserved word in BigQuery standard SQL. Using it as a dataset name requires backtick-quoting in every generated query; the dbt BigQuery adapter handles this in some versions but not all ‚Äî behaviour is version-dependent and not documented as a guarantee. Renaming eliminates the ambiguity at zero cost pre-implementation. Affects: `meltano.yml` target dataset config, `dbt_project.yml`, `sources.yml`, Dagster `DbtProject` config, `scripts/generate_parquet.py` (default `--dataset` argument), notebooks. BRD updated to v2.6; REQ-003.1 Open Item 2 closed. |
| 2026-03-14 | Meltano tap plugin | Switched from `tap-spreadsheets-anywhere` to `tap-csv` (MeltanoLabs variant) | `tap-csv` handles all 9 CSVs correctly including the UTF-8 BOM on `product_category_name_translation.csv` (no `encoding: utf-8-sig` needed ‚Äî Python csv module strips BOM transparently). 1M-row geolocation file loaded in ~31s. Simpler config structure (`entity`/`path`/`keys` vs `tables` array). |
| 2026-03-14 | Meltano target method | Changed `target-bigquery` method from `storage_write_api` to `batch_job` | `storage_write_api` holds open gRPC streams per table. When `tap-csv` processes files sequentially, earlier streams go idle during the 31s geolocation load and hit BigQuery's 600s inactivity timeout ‚Äî `olist_order_items_dataset` stream was killed. `batch_job` uses traditional load jobs with no persistent streams, immune to idle timeouts. Trade-off: slower (~34 min total) but reliable for sequential taps. |
| 2026-03-14 | BigQuery table naming ‚Äî `_view` suffix | `target-bigquery` with `denormalized: false` stores data as JSON blobs in base tables and creates flat-column views with `_view` suffix. dbt `sources.yml` must reference the view names (e.g. `olist_customers_dataset_view`) instead of the original stream names. | **DOWNSTREAM IMPACT (Agent 1b):** `sources.yml` table names must ALL have `_view` suffix appended. Full mapping: `olist_customers_dataset_view`, `olist_orders_dataset_view`, `olist_order_items_dataset_view`, `olist_order_payments_dataset_view`, `olist_order_reviews_dataset_view`, `olist_products_dataset_view`, `olist_sellers_dataset_view`, `olist_geolocation_dataset_view`, `product_category_name_translation_view`. Base tables contain a `data` JSON column ‚Äî do NOT query base tables directly. Also applies to Dagster `AssetKey` values (Agent 1c) and `AssetSpec` names. |
| 2026-03-14 | dbt staging ‚Äî `sources.yml` adapted to `_view` suffix | All 9 `sources.yml` table names use `_view` suffix to match `target-bigquery` `denormalized: false` output. Spec originally listed names without suffix. | Upstream deviation from Agent 1a (changelog 2026-03-14 `_view` suffix entry). `sources.yml` is the naming contract ‚Äî all staging models use `source('olist_raw', '..._view')`. No functional impact on downstream marts (they reference `ref('stg_*')`, not raw sources). |
