**Key message:** 6 modular layers, each independently replaceable.

### Speaker Notes
- Walk left-to-right through the flow: CSV → Meltano → BigQuery raw → dbt → BigQuery analytics → Jupyter → Parquet → Streamlit.
- **Automated boundary** (orange line): "Everything from ingestion through dbt testing is orchestrated by Dagster — runs daily at 09:00 SGT or on manual trigger."
- **Manual boundary** (purple line): "Notebooks, Parquet export, and the dashboard are run by the analyst. The Parquet files act as a decoupling layer — the Streamlit dashboard requires zero GCP credentials."

## Slide 4 — Tool Selection Rationale

**Key message:** Each tool was chosen for a specific architectural property — decoupling is the design principle.

### Speaker Notes
- **Meltano:** "We needed declarative, reproducible ingestion. The Singer protocol separates extraction from loading — we can swap the target without touching the tap config. `WRITE_TRUNCATE` ensures idempotent full refresh on every run."
- **dbt:** "SQL-first transformation with `ref()` for lineage. All type casting happens in the staging layer — marts never touch raw strings. This is ELT, not ETL."
- **dbt-expectations:** "76 tests integrated directly into the DAG. A failing test blocks downstream marts. We calibrated thresholds from source data profiling — `mostly=0.08` for review comment titles, `mostly=0.97` for geolocation."
- **Dagster:** "Asset-centric, not task-centric. `dagster-dbt` auto-generates one Dagster asset per dbt model — the lineage graph is derived, not maintained."
- **Streamlit:** "Python-native, reads Parquet files. No database connection required for the dashboard layer."
- **Why not alternatives?** "Custom Python scripts lack lineage. Airflow is DAG-centric, not asset-centric. Looker requires a persistent database connection."
- Transition: "With these tools chosen, here's the schema we designed."

## Slide 5 — Star Schema Design

**Key message:** 3 fact tables at different grains + 4 dimension tables — single joins for any analytical query.

### Speaker Notes
- **Left column — Facts:**
  - `fct_sales`: "Order-item grain, ~112k rows. The most complex model — it's a 3-source CTE joining items → orders → customers. You cannot join items directly to customers because `customer_unique_id` is only in `stg_customers`, reachable through `stg_orders.customer_id`."
  - `fct_reviews`: "Review grain. 789 source duplicates removed via `ROW_NUMBER`. The FK points to `stg_orders`, not `fct_sales` — 756 orders have reviews but no line items. This is documented in ADR-003."
  - `fct_payments`: "Payment-method grain. Date key comes from `stg_orders` via LEFT JOIN — nullable when the order is missing."
- **Right column — Dims:**
  - `dim_customers`: "Deduped on `customer_unique_id`. Enriched with geolocation lat/lng."
  - `dim_products`: "Category name resolved via COALESCE — English → Portuguese → 'uncategorized'. 610 products had null categories."
  - `dim_date`: "Generated by `dbt_utils.date_spine`. DATE type — see ADR-001 for why we chose DATE over INTEGER."
- **Bottom rationale:** "Star schema gives us single-hop joins, independent dimension evolution, and clear grain per fact table. The staging layer fixes all data defects once — every consumer inherits clean data."
- Transition: "That clean data is enforced by our testing framework."

## Slide 8 — Data Quality Framework

**Key message:** 76 tests, 2 mechanisms, 0 failures.

### Speaker Notes
- **Generic tests (left):** "Schema-level tests declared in YAML. PKs get `not_null` + `unique`. FKs get `relationships`. We use `dbt-expectations` for range checks, accepted values, and row count guards."
- **Singular tests (right):**
  - "Boleto test: Every boleto payment must have exactly 1 instalment — this only passes after the staging layer clamps `0 → 1`."
  - "Reconciliation test: Verifies `SUM(payment_value) ≈ SUM(total_sale_amount)` per order within R$1 tolerance."
  - "Date range test: All fact table `date_key` values fall within 2016-01-01 to 2018-12-31."
- **Staging fixes table:** "All corrections happen at the staging layer. 789 duplicate reviews, ~1M geolocation rows filtered to ~19k via bounding box, 3 'not_defined' payments removed, 2 zero-instalment records clamped, and 610 uncategorised products labelled."
- Transition: "With clean data in hand, here are the market insights."

## Slide 9 — Architecture Decisions & Lessons Learned

**Key message:** Four formal ADRs + non-obvious implementation patterns that shaped the pipeline.

### Speaker Notes
- **ADR-001:** "`date_key` as DATE, not INTEGER. `dbt_utils.date_spine` natively produces DATE — using INTEGER would add unnecessary casting to every model that touches a date."
- **ADR-002:** "Dataset names `olist_raw` and `olist_analytics`. The original name 'raw' is a BigQuery SQL reserved word — it requires backtick-quoting in every query."
- **ADR-003:** "`fct_reviews.order_id` → `stg_orders`, not `fct_sales`. 756 orders have reviews but zero line items. Pointing the FK to `fct_sales` would orphan those reviews."
- **3-source CTE:** "This tripped us up early. Joining `stg_order_items` directly to `stg_customers` produces zero matches because `customer_unique_id` is only reachable through `stg_orders.customer_id`. The CTE chain is: items → orders (get customer_id) → customers (get customer_unique_id)."
- **batch_job method:** "The 1M-row geolocation file caused gRPC idle timeouts in BigQuery's Write API streaming mode. Switching to `batch_job` resolved it."
- **dbt-expectations constraint:** "The `mostly` parameter — which lets you set a threshold like 'at least 97% of values must pass' — is documented in dbt-expectations but not implemented in the Metaplane fork (v0.6.0). Four proportion-based tests had to be removed or restructured."
- Transition: "Finally, let me explain how we built all of this with AI agents."

## Slide 10 — AI Multi-Agent Architecture

**Key message:** Contract-driven isolation enables parallel AI agents to build a coherent pipeline.

### Speaker Notes
- **Agent table:** Walk through each agent's scope:
  - "Agent 1 (Data Engineer) ran in 4 phases: Meltano config, staging models, mart models, and dbt tests. 76 tests passing."
  - "Agent 2 (Platform Engineer) built the Dagster orchestration layer — schedule, launch scripts, `.env` integration."
  - "Agent 3 (Analytics Engineer) produced 4 Jupyter notebooks with 11 metrics and 6 Parquet exports."
  - "Agent 4 (Dashboard Engineer) built the 5-page Streamlit dashboard with 4 global filters."
  - "Agent 5 (Data Scientist) created the executive brief, slide deck, and speaker notes."
  - "The Orchestrator — that's me, using Claude Opus — managed gate dependencies, resolved merge conflicts, produced diagrams, and wrote the technical report."
- **Principles:**
  - "Isolation: Each agent works in a separate git worktree. They never see each other's uncommitted work."
  - "Dependency gates: GATE-1 (Meltano config) → GATE-2 (dbt models) → GATE-3 (Parquet files) → GATE-4 (dashboard). Each gate is a merge to main that the orchestrator validates."
  - "Contract surfaces: CLAUDE.md is the specification. `sources.yml` is the contract between Meltano and dbt. Parquet column schemas are the contract between notebooks and dashboard."
  - "42 changelog entries track every deviation from the BRD."
- **Closing:** "This architecture let us build a production-grade pipeline where each agent focused on its domain while the orchestrator ensured coherence across the whole system."
