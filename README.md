# Project Caravela — Olist E-Commerce Analytics Pipeline

End-to-end data pipeline for the [Brazilian E-Commerce Dataset by Olist](https://www.kaggle.com/datasets/olistbr/brazilian-ecommerce) (~100k orders, 2016–2018). Ingests 9 CSV source files into BigQuery, transforms them through a star-schema data warehouse with dbt, orchestrates execution with Dagster, analyses the data in Jupyter notebooks, and presents insights through a Streamlit dashboard.

---

## Architecture Overview

```
raw_data/ (9 CSVs)
    └── Meltano (tap-spreadsheets-anywhere → target-bigquery)
            └── BigQuery olist_raw (9 tables)
                    └── dbt (staging views → mart tables)
                            └── BigQuery olist_analytics (4 dims + 3 facts)
                                    └── Jupyter Notebooks (analysis + Parquet export)
                                                └── Streamlit Dashboard (reads Parquet)
```

Ingestion through dbt testing is orchestrated end-to-end by Dagster with a daily 09:00 SGT schedule. The dashboard reads pre-committed Parquet files — no live BigQuery connection required.

## Quickstart

### Prerequisites

```bash
# 1. Create conda environment
conda env create -f environment.yml
conda activate assignment2

# 2. Copy and fill in credentials
cp .env.example .env
# Edit .env: set GCP_PROJECT_ID and GOOGLE_APPLICATION_CREDENTIALS

# 3. Pre-create BigQuery datasets (one-time, platform engineer)
bq mk --dataset <project>:olist_raw
bq mk --dataset <project>:olist_analytics
```

### Data Pipeline (BigQuery required)

```bash
# Install Meltano plugins (one-time per machine)
cd meltano && meltano install && cd ..

# Run ingestion: 9 CSVs → BigQuery olist_raw
cd meltano && meltano run tap-spreadsheets-anywhere target-bigquery && cd ..

# Install dbt packages and compile manifest
cd dbt && dbt deps && dbt parse

# Build all models + run all tests (interleaved — a failing test blocks downstream marts)
dbt build && cd ..

# Start Dagster UI + daemon (requires manifest.json from dbt parse above)
source .env
dagster dev
```

### Dashboard (no BigQuery required)

Parquet files are committed to `streamlit/data/` — the dashboard runs directly from those:

```bash
conda activate assignment2
streamlit run streamlit/dashboard.py
```

Opens at `http://localhost:8501`.

### Regenerate Parquet files from BigQuery

If you want to rebuild the Parquet exports without running the notebooks:

```bash
source .env
python scripts/generate_parquet.py
# Or with explicit project:
python scripts/generate_parquet.py --project <gcp_project_id>
```

## Transformation

dbt transforms raw strings into a typed star schema in `olist_analytics`. All source data type-casting, deduplication, and defect corrections happen at the **staging layer** — mart models consume clean views.

### Models

| Layer | Models | Materialisation |
|---|---|---|
| Staging | `stg_customers`, `stg_orders`, `stg_order_items`, `stg_payments`, `stg_reviews`, `stg_products`, `stg_sellers`, `stg_geolocation`, `stg_product_category_name_translation` | BigQuery views |
| Marts — Dimensions | `dim_customers`, `dim_products`, `dim_sellers`, `dim_date` | BigQuery tables |
| Marts — Facts | `fct_sales`, `fct_reviews`, `fct_payments` | BigQuery tables |

### Star schema

- **`fct_sales`** — order-item grain; 4 FKs (customers, products, sellers, date); `total_sale_amount = price + freight_value`
- **`fct_reviews`** — review grain; PK deduped (789 source dupes removed); FK → `stg_orders` (not `fct_sales`)
- **`fct_payments`** — payment-method grain; compound PK: `order_id + payment_sequential`
- **`dim_date`** — generated by `dbt_utils.date_spine` (2016-01-01 to 2018-12-31)

### Running dbt

```bash
cd dbt
dbt deps          # install dbt-expectations + dbt_utils packages
dbt parse         # compile manifest.json (required before dagster dev)
dbt build         # run all models + all tests in topological order
dbt docs serve    # optional: interactive schema + lineage browser on :8080
```

### Data quality tests

76 tests across generic (schema.yml) and singular (SQL files) mechanisms, 0 failures:

- **Generic:** `not_null`, `unique`, `accepted_values`, `relationships`, `expect_column_values_to_be_between`, `expect_table_row_count_to_be_between`, `unique_combination_of_columns`, temporal pair tests
- **Singular:** `assert_boleto_single_installment.sql`, `assert_payment_reconciliation.sql`, `assert_date_key_range.sql`

## Orchestration

Dagster orchestrates the full pipeline as an asset graph. `dagster-dbt` auto-generates one Dagster asset per dbt model from `manifest.json`. The Meltano ingestion step is wired as an upstream asset so the lineage graph is complete.

### Assets

| Asset | Type | Description |
|---|---|---|
| `meltano_ingest` | Shell asset | Runs `meltano run tap-spreadsheets-anywhere target-bigquery` |
| `stg_*` (9 assets) | dbt model assets | Staging views — materialised after `meltano_ingest` |
| `dim_*`, `fct_*` (7 assets) | dbt model assets | Mart tables — materialised after all staging models pass tests |

### Schedule

Daily at **09:00 SGT** (`Asia/Singapore` timezone), plus manual trigger via the Dagster UI.

### Running Dagster

```bash
# Set credentials (required before dagster dev)
source .env

# Start Dagster — webserver + daemon in one command
dagster dev
```

Open `http://localhost:3000`. Navigate to **Assets → Materialize all** for a manual run, or let the schedule trigger automatically.

**Pre-requisite:** `manifest.json` must exist before starting Dagster. If the file is missing:

```bash
cd dbt && dbt parse && cd ..
```

## Data Quality Notes

- **Observation window**: Jan 2017 – Aug 2018 (20 months). 2016-11/12 (0–1 orders) and 2018-09/10 (16/4 orders) excluded as data cut artefacts in all trend analyses.
- **Product categories**: Always use `product_category_name_english` — `product_category_name` contains empty strings for 610 products.
- **Delivery metrics**: Use `COUNT(DISTINCT order_id)` — delivery timestamps are order-level attributes repeated across item rows in `fct_sales`.
- **RFM reference date**: Hardcoded `2018-08-31` for reproducibility — not `CURRENT_DATE`.
- **Geolocation**: Filtered to Brazil bounding box before aggregation; lat/lng is nullable on `dim_customers` and `dim_sellers` (~3% of records have no geolocation match).
